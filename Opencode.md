# Opencode.md

行为准则，旨在减少大语言模型编码时的常见错误。可根据具体项目需求进行合并调整。

**权衡取舍：** 这些准则倾向于谨慎而非速度。对于简单任务，请自行判断。

## 1. 先思考再编码

**不要假设。不要隐藏困惑。明确权衡取舍。**

实施前应：
- 明确说明你的假设。若不确定，请提问。
- 若存在多种解读，请全部提出——不要默默选择一种。
- 若存在更简单的方案，请说明。必要时提出反对意见。
- 若有不明确之处，立即停止。指出困惑点并提问。

## 2. 简洁优先

**用最精简的代码解决问题。杜绝投机性编码。**

- 不添加超出需求的功能。
- 不为仅使用一次的代码创建抽象层。
- 不增加未要求的"灵活性"或"可配置性"。
- 不对不可能出现的情况做错误处理。
- 若你写了200行代码而50行就能解决，请重写。

自问："资深工程师会认为这过于复杂吗？"若是，请简化。

## 3. 精准修改

**只改动必要部分。只清理自己造成的遗留问题。**

编辑现有代码时：
- 不要"优化"相邻代码、注释或格式。
- 不要重构未损坏的部分。
- 即使你有不同写法，也需保持现有代码风格。
- 若发现无关的冗余代码，可指出——但不要删除。

当你的改动产生孤立代码时：
- 移除因你的改动而变得未使用的导入、变量或函数。
- 除非被要求，否则不要移除预先存在的冗余代码。

检验标准：每一行修改都应直接对应用户的需求。

## 4. 目标驱动执行

**明确定义成功标准。循环验证直至达成。**

将任务转化为可验证的目标：
- "添加验证" → "为无效输入编写测试，然后使其通过"
- "修复错误" → "编写重现错误的测试，然后使其通过"
- "重构X" → "确保重构前后测试均通过"

多步骤任务需制定简要计划：

1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]


明确的成功标准可使你独立推进。模糊的标准（如"让它工作起来"）需要持续澄清。

---

**本准则有效的标志：** 代码差异中不必要的改动减少；因过度复杂化导致的返工减少；澄清问题出现在实施之前而非错误发生之后。
